# Terniq V1 Design

## Positioning

Terniq is a Codex-native workflow product for serious software work.

It is not a generic prompt pack, and it is not a thin rename of an older host-specific system. Its purpose is to package durable engineering habits into Codex-native surfaces:

- `skills` for judgment-heavy workflows
- `agents` for parallel specialist reasoning
- `commands` for explicit entry points
- `.codex-plugin` for distribution and discovery

The product should feel like it was designed for Codex from the start.

## Product Thesis

Terniq should help a user move through the real software loop:

`decide -> build -> verify -> ship`

without collapsing the entire process into one vague assistant persona.

The product wins if it makes Codex behave more like a disciplined engineering partner:

- clearer scoping before implementation
- stronger root-cause debugging
- sharper review before merge
- more intentional frontend design decisions
- cleaner source gathering and research flow
- stronger writing and editing quality
- better use of Codex-native primitives such as subagents, browser verification, plugins, apps, and automations

## Problem

Most workflow packs mix two different layers:

1. The valuable layer: engineering judgment, review discipline, debugging method, verification rigor.
2. The accidental layer: host-specific assumptions about prompts, hooks, fetch mechanisms, marketplace behavior, and tool availability.

That coupling makes "compatibility" look easier than it is. A pack can install on Codex and still feel non-native if the host model in its bones is from somewhere else.

Terniq exists to separate method from host. The habits are portable. The execution substrate is not.

## Non-Goals

- Not a general-purpose AI persona.
- Not a clone of Waza with renamed files.
- Not a broad MCP platform in V1.
- Not an app-connector-first product in V1.
- Not a giant bag of dozens of tiny skills.
- Not a replacement for Codex's built-in tools.

## Codex-Native Design Principles

### 1. Host realism over cross-platform purity

Terniq should optimize for the real Codex environment:

- local workspace access
- plugin manifests
- skill discovery
- subagents
- browser-use style verification
- app connectors where relevant
- automations where repeated workflows justify scheduling

If a workflow is stronger because Codex exposes a native primitive, that is a feature, not a compromise.

### 2. Judgment stays in skills

Anything that depends on interpretation, pressure-testing, prioritization, or taste belongs in a skill.

Examples:

- whether a feature should exist
- whether a change is scope drift
- whether a bug hypothesis is testable
- whether a design direction is specific enough
- whether a source is primary enough to trust

### 3. Deterministic execution gets pushed down

Anything repetitive, explicit, and checkable should move lower when the product matures:

- commands
- helper scripts
- plugin wrappers
- browser verification flows

The goal is fewer bloated skills and cleaner boundaries between reasoning and execution.

### 4. Verification is part of the product

Terniq should not let "done" mean "probably correct."

Every high-value workflow ends with evidence:

- tests
- browser checks
- diff review
- live command output
- explicit unresolved risk when verification is blocked

### 5. Small set, deep behaviors

The product should ship a restrained number of broad, high-value workflows rather than many narrow leaf skills.

Eight skills is acceptable because the reference shape is already proven. The constraint is not skill count. The constraint is whether each skill owns a real workflow boundary.

## V1 Release Strategy

V1 should be the full first release, not a half-formed MVP.

The reference implementation already proves that these workflows have real value. The work in front of us is not invention from zero; it is host-native redesign.

So V1 should ship the full Terniq workflow suite on day one:

- `terniq-think`
- `terniq-design`
- `terniq-check`
- `terniq-hunt`
- `terniq-read`
- `terniq-learn`
- `terniq-write`
- `terniq-health`

The product should still stay narrow by keeping these eight workflows deep and opinionated instead of proliferating variants.

## V1 Product Surface

### Skills

#### 1. `terniq-think`

Purpose:

- plan before implementation
- evaluate whether work should exist
- pressure-test design assumptions
- force one recommended direction instead of broad option dumping

Codex-native behavior:

- inspects the actual repo before opining
- uses official docs or primary sources when the question is current or framework-specific
- names the most fragile assumption explicitly
- stops before implementation

#### 2. `terniq-design`

Purpose:

- produce intentional UI direction instead of generic defaults
- handle screenshot-driven iteration
- enforce browser verification for visible work

Codex-native behavior:

- treats Browser Use as the preferred visual verification surface
- distinguishes app-shell work from marketing-surface work
- requires a named aesthetic direction before major UI work
- includes mobile-width verification in the workflow

#### 3. `terniq-check`

Purpose:

- review diffs before merge
- classify findings by severity and actionability
- route deeper reviews to specialists
- require real verification evidence

Codex-native behavior:

- uses subagents for parallel specialist passes when diff depth justifies it
- treats app/tool/browser verification as first-class evidence when relevant
- supports both diff review and issue/PR triage mode
- keeps safe autofix and gated changes separate

#### 4. `terniq-hunt`

Purpose:

- debug broken behavior, failing tests, and integration/tool failures
- confirm root cause before any fix

Codex-native behavior:

- forces hypothesis quality
- treats host/tool failures as diagnostic problems, not excuses to switch methods blindly
- prefers evidence from the real runtime over guessed architecture

#### 5. `terniq-read`

Purpose:

- fetch and normalize external material
- convert URLs or PDFs into usable working material
- stop before interpretation unless the user asked for it

Codex-native behavior:

- uses Codex-native `web` and connected surfaces first where appropriate
- falls back only when a specific source needs a different route
- preserves storage/reporting discipline
- remains a content-ingestion workflow, not a research summary workflow

#### 6. `terniq-learn`

Purpose:

- turn unfamiliar domains or collected sources into structured understanding
- support research-to-writing workflows

Codex-native behavior:

- depends on `terniq-read` for collection
- uses web search only for discovery, not as a substitute for source reading
- can use specialist research agents in deeper synthesis mode
- separates quick-reference mode from publish-ready mode

#### 7. `terniq-write`

Purpose:

- remove AI stiffness from prose
- rewrite or refine writing without changing meaning
- support CN/EN editing with audience awareness

Codex-native behavior:

- works as an editing surface, not a generic "please write everything" personality
- keeps structure stable unless explicitly asked to restructure
- supports bilingual validation and release-note formatting

#### 8. `terniq-health`

Purpose:

- audit the Codex operating environment itself
- identify where environment drift breaks behavior or reliability

Codex-native behavior:

- audits `AGENTS.md`, local skill layout, plugin enablement, marketplace presence, connector visibility, automation usage, available tools, and verification discipline
- does not assume Claude-style hooks or CLAUDE.md layering
- distinguishes missing data from real findings

### Agents

V1 agents should stay few and high leverage.

Core agents:

- `reviewer-security`
- `reviewer-architecture`
- `reviewer-frontend`
- `researcher`
- `environment-auditor`

Agent usage rules:

- agents are sidecars, not the main product
- `terniq-check` owns review orchestration
- `terniq-learn` owns research orchestration
- `terniq-health` owns environment audit orchestration
- commands do not duplicate agent logic; they only trigger the owning workflow

### Commands

Commands exist for explicit entry, not because latent skill discovery is weak.

V1 commands:

- `/terniq:plan`
- `/terniq:design`
- `/terniq:review`
- `/terniq:debug`
- `/terniq:read`
- `/terniq:research`
- `/terniq:edit`
- `/terniq:health`

Command rules:

- every command maps to exactly one owning skill
- commands remain thin wrappers, not separate workflow trees
- explicit commands are especially useful when users want deterministic invocation instead of discovery

## Activation Model

Terniq should support both discovery and explicit invocation.

### Discovery path

The plugin relies on Codex skill discovery through strong metadata:

- clear descriptions
- real trigger phrases
- strong boundaries
- host-specific wording where useful

### Explicit path

The plugin exposes commands for users who want a guaranteed workflow entry:

- planning
- design
- review
- debugging
- reading
- research
- writing
- environment audit

### Escalation path

Some skills should escalate internally based on real conditions:

- `terniq-check`: base review -> specialist review -> adversarial review
- `terniq-learn`: quick reference -> deep research
- `terniq-design`: screenshot iteration -> full direction lock
- `terniq-health`: local audit -> multi-agent environment audit

## Plugin Manifest Requirements

The plugin shell should be minimal and real.

Required in `.codex-plugin/plugin.json`:

- `name: terniq`
- versioned release metadata
- clear product description
- `skills: "./skills/"`
- meaningful interface metadata

Recommended interface intent:

- category: `Coding`
- capabilities: `Interactive`, `Read`, `Write`
- prompt examples that express real workflows rather than slogans

Example prompt directions:

- "Review this diff before I merge it"
- "Help me debug this failing integration"
- "Plan this feature before we implement it"

## Skill Naming Decision

Directory names and visible names should keep the `terniq-` prefix in V1.

Reason:

- avoids collisions with built-in or third-party skills
- makes discovery behavior easier to debug
- keeps plugin identity explicit during early adoption

This can be relaxed later if Codex plugin namespacing proves strong enough.

## Workflow Contracts

### `terniq-think`

Input:

- rough product idea
- architecture decision
- "should we keep this"
- "what is the right way to implement this"

Output:

- one recommended direction
- one alternative only when genuinely close
- explicit fragile assumption
- scope boundary
- verification plan before implementation begins

### `terniq-design`

Input:

- UI request
- component/page design work
- screenshot-based complaint
- visual redesign

Output:

- named design direction
- direction constraints
- implementation guidance
- browser verification requirement

### `terniq-check`

Input:

- diff
- PR
- issue triage request
- "review this"

Output:

- scope drift status
- severity-ranked findings
- safe autofix vs gated changes
- specialist review when triggered
- explicit verification status with evidence

### `terniq-hunt`

Input:

- error
- failing test
- broken behavior
- external tool failure that needs diagnosis

Output:

- specific hypothesis
- evidence trail
- confirmed root cause
- only then a fix direction

### `terniq-read`

Input:

- URL
- PDF
- "fetch this"
- "save this page"

Output:

- normalized content
- saved path or retrieved body
- no analysis unless explicitly requested

### `terniq-learn`

Input:

- unfamiliar topic
- source bundle
- "research this"
- "turn these materials into understanding"

Output:

- structured notes or draft
- source-backed outline
- explicit mode: quick reference or deep research

### `terniq-write`

Input:

- draft text
- editing request
- bilingual consistency request
- release note request

Output:

- rewritten prose only
- no extra explanation unless asked

### `terniq-health`

Input:

- "audit my Codex setup"
- "why is Codex behaving weirdly"
- plugin or connector visibility issues
- environment drift questions

Output:

- Codex-specific health report
- findings grouped by severity
- concrete fix actions

## Differentiation From Waza

Terniq should preserve the good instincts and replace the wrong substrate.

What to preserve:

- narrow, high-value workflows
- strong opinions
- engineering rigor
- explicit workflow boundaries

What to redesign:

- remove Claude-centric language and assumptions
- treat subagents as a first-class review and research primitive
- treat Browser Use, plugins, apps, and automations as real product surface
- rebuild health around Codex reality rather than Claude hooks
- shift source reading toward Codex-native web access instead of host-specific fetch folklore

## Release Scope

V1 should ship all of the following:

- plugin manifest
- eight skills
- five agents
- eight commands
- root README
- one design doc

That is the smallest release that actually matches the product thesis.

Shipping less would turn Terniq into a partial demo. Shipping this set keeps the release coherent.

## Risks

### Risk 1: Too much overlap with existing public skills

If Terniq sounds like a renamed bundle of other skills, it loses identity.

Mitigation:

- keep host-native execution rules sharp
- make boundaries explicit
- make commands and plugin packaging coherent

### Risk 2: Full-release scope creates loose quality

The danger is not the number of workflows. The danger is shallow rewriting.

Mitigation:

- keep the workflow count at eight
- rewrite each one around Codex-native behavior instead of doing search-and-replace adaptation

### Risk 3: Commands and skills drift apart

If commands become independent logic trees, maintenance gets ugly quickly.

Mitigation:

- commands remain wrappers over the owning workflow contracts

### Risk 4: `terniq-health` becomes too broad

Environment auditing can balloon into platform archaeology.

Mitigation:

- keep `terniq-health` scoped to behavior-affecting layers:
  - instructions
  - skills
  - plugins
  - connectors
  - automations
  - verification

### Phase 4: Fit and validation

- test discovery quality in Codex
- test command invocation quality

- refine descriptions and activation boundaries

## Open Questions

These are the remaining real design questions after scope lock:

1. Should `terniq-read` save by default or preview by default inside Codex-native web workflows?
2. How aggressive should `terniq-check` be about auto-invoking Browser Use when frontend files are touched?
3. Should `terniq-learn` and `terniq-write` stay separate commands if users often treat them as one chain?

## Decisions For Now

- `terniq-read` keeps save/report discipline by default.
- `terniq-check` should recommend browser verification for frontend changes and make it mandatory only when the user asked for UI validation or when the change is clearly user-visible.
- `terniq-learn` and `terniq-write` remain separate because synthesis and polishing are different workflows even when often chained.

This is the right first version because it is broad enough to feel like a real product and still constrained enough to keep each surface legible.

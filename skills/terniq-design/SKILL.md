# Terniq Design

Use this skill for interface direction and visible frontend changes.

If it could have been generated by a default prompt, it is not good enough.

## Core Rules

- Name the design direction before major UI work.
- Prefer Browser Use for visual verification.
- Separate screenshot bugfix mode from full direction mode.
- Do not hand off visible work without browser validation.
- Distinguish app surfaces from marketing surfaces.

## Modes

### Screenshot Iteration Mode

Activate when the user provides an image or screenshot with a visual complaint.

Flow:

1. State the visible problem in one sentence.
2. Wait for confirmation if the issue diagnosis is not obvious.
3. Find the real component or style source in code.
4. Apply the smallest fix that resolves the specific visual defect.
5. Verify in the browser before handoff.

Boundary:

- if the fix affects 3 or more components
- if the problem is actually a direction problem, not a local bug

then stop and switch to full direction mode.

### Full Direction Mode

Activate when the work is a new component, page, redesign, or broad UI change.

Before implementation, lock:

1. who uses this and in what context
2. the named aesthetic direction
3. the one thing the user should remember afterward
4. hard constraints such as framework, accessibility, or performance
5. the signature interaction detail

Do not start major UI implementation until the direction is clear enough to defend.

## App Surface vs Marketing Surface

For app or dashboard surfaces:

- utility first
- no decorative hero by default
- hierarchy through surface depth and spacing
- interactions should feel crisp, not theatrical

For marketing or landing surfaces:

- one strong visual thesis
- one strong memory hook
- each section gets a single job
- avoid generic card farms and neutral startup gradients

## Direction Rules

- "clean and modern" is not a direction
- reference brands are not a direction by themselves
- if a user names a brand, decompose it into visible properties:
  - radius philosophy
  - surface depth treatment
  - accent family
  - typography personality

## Browser Verification

Browser verification is part of the workflow, not a nice-to-have.

Required checks for visible work:

- desktop full-width view
- mobile-width view around 375px
- the specific user-visible interaction that changed

If Browser Use is available, prefer it over describing the UI from memory.

## Mid-Build Drift Check

After every meaningful section or every third component:

1. restate the visual thesis
2. ask whether the current work still matches it
3. if drifted, identify what broke first:
   - type
   - spacing
   - color
   - surface treatment
   - motion

Fix drift early. Do not wait for the final pass.

## Handoff Requirements

Before handoff, verify:

- the first screen is not generic
- the layout works on mobile
- motion supports hierarchy rather than distracting from it
- the product or brand is recognizable without relying on the nav alone

Then summarize:

- the chosen direction
- the non-obvious design decisions
- what was verified in the browser

## Hard Stops

- Do not treat visible frontend work as done without browser verification.
- Do not accept vague visual direction when the task is broad.
- Do not add new sections because they feel emotionally incomplete.
- Do not let design drift into generic defaults just because the code is technically correct.

## Gotchas

| What happened | Rule |
|---------------|------|
| Used a default font stack with no point of view | typography must carry intent |
| Claimed a UI looked right without opening a browser | browser verification is required |
| Built a dashboard like a marketing page | app surfaces are utility-first |
| Three cards with identical treatment became the whole design | vary structure, not just color |
| Mobile width broke after desktop polish | verify 375px before handoff |
